# CoreTab 测试用例文档 (v2.1)

> 新标签页替代 + 标签管理 + Closed Tabs + GitHub Trending

## 1. 测试环境

| 项目 | 要求 |
|------|------|
| 浏览器 | Chrome 88+ (Manifest V3) |
| 操作系统 | Windows/macOS/Linux |
| 加载方式 | 开发者模式加载已解压的扩展程序 |

### 1.1 扩展加载步骤
1. 打开 Chrome，访问 `chrome://extensions/`
2. 开启「开发者模式」
3. 点击「重新加载」CoreTab 扩展
4. 打开新标签页验证

### 1.2 页面区域顺序
> 固定顺序规则，不随功能增减而改变

1. Open Tabs
2. Closed Tabs
3. GitHub Trending
4. History

## 6. 经验沉淀（开发教训）

### 6.1 HTML/JS 元素 ID 同步规则
> **问题**: 修改 UI 结构时，只改了 HTML 没改 JS，或只改了 JS 没改 HTML，导致功能失效
>
> **教训**:
> - 任何 DOM 元素 ID 变更必须在 HTML 和 JS 中同时修改
> - 重构前先搜索所有引用位置
> - 涉及 ID 变更时，优先保持旧 ID 或创建新的
>
> **案例**: GitHub Trending 的 `githubHotCard` 改为 `githubGrid`，但 `loadGitHubTrending()` 函数还在查找旧 ID，导致一直 Loading

### 6.2 重复事件监听器
> **问题**: 两个 `document.addEventListener('click', ...)` 导致事件冲突
>
> **教训**:
> - 整个应用只允许一个全局 click 监听器
> - 使用事件委托（event delegation）
> - 统一在底部注册，移除散落的监听器

### 6.3 关闭标签时记录 Closed Tabs
> **问题**: 关闭标签后没有记录，导致 Closed Tabs 功能缺失
>
> **教训**:
> - 任何关闭标签的操作（单关、域名关、全关）都需要调用 `addClosedTab()`
> - 关闭前记录，而不是关闭后记录

### 6.4 页面区域顺序规则
> **规则**: Open Tabs → Closed Tabs → GitHub Trending → History
>
> **教训**:
> - 区域顺序固定，不随功能增减而改变
> - 新增功能必须遵守此顺序
> - 文档和代码必须同步更新

### 6.5 存储 Key 命名规范
> **规则**: 使用 `coretab_` 前缀
>
> **教训**:
> - 避免与 tab-out 等其他扩展冲突
> - 缓存数据加版本号或功能前缀
